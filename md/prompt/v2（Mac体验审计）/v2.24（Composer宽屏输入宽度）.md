# v2.24 / Composer 宽屏输入宽度

## 版本号和分配依据

- 版本号：`v2.24`
- 分配依据：`update_log.md` 最新正式版本为 `v2.23 / 运行策略开关宽屏网格`，本轮继续递增到 `v2.24`。
- 所属总目标：Agent X 持续优化 UI、Mac 和 iPad 体验。

## 背景

v2.22 已经为聊天气泡建立宽屏阅读宽度策略，避免 Mac/iPad 宽窗口中消息文本过窄或过长。但推理页底部 `ComposerBar` 仍只按 `chatSurface` 横向 padding 展开，在 iPad 和 Mac Catalyst 宽窗口中输入框会横跨过长距离，prompt 编辑行长不稳定，也和聊天气泡的阅读宽度不一致。

只读子 agent 审计确认 `ChatWorkspace.chatSurface` 直接堆叠 `ChatTranscript` 和 `ComposerBar`，composer 发送、停止、焦点、`Command+Return` 和模板填入/发送语义都封装在现有 `ComposerBar`、`ComposerInputMetadata`、`ComposerFocusPolicy` 和 `InferenceEngine` 状态流中。本轮应只在外层加宽屏布局策略，不改业务状态。

## 目标

1. 新增 `ComposerBarLayoutPolicy`，集中定义 composer 横向 padding、底部 padding、最小可读宽度和最大内容宽度。
2. 在 `ChatWorkspace.chatSurface` 中让 `ComposerBar` 在宽屏居中并限制最大内容宽度。
3. iPhone 和窄 split view 保持原有可用宽度；iPad/Mac 宽区域不再让 composer 无限拉伸。
4. 保留 `ComposerBar` 内部 `TextField`、发送/停止 `Button`、`@FocusState`、`Command+Return` 和现有辅助语义。
5. 新增 XCTest 锁住 composer 宽屏输入宽度策略。
6. 同步 `AGENTS.md`、`README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和 `update_log.md`。

## 非目标

- 不修改 `InferenceEngine.send`、`stop`、`applyTemplate` 或 `useTemplate` 状态流。
- 不修改 `ComposerInputMetadata` 的 label/value/hint/input labels/identifier。
- 不修改 `ComposerFocusReason`、`ComposerFocusRequest` 或 `ComposerFocusPolicy`。
- 不修改聊天消息气泡、聊天记录容器、会话创建/切换/删除、导出、模型、artifact、runtime、设置或提示词模板内容。
- 不新增 UI Test target。
- 不下载模型权重，不接入云端推理，不接入真实 runtime，不创建原生 macOS target。

## 当前架构依据

- `LocalGemma/ContentView.swift`
  - `ChatWorkspace.chatSurface` 垂直堆叠 `ChatTranscript` 和 `ComposerBar`。
  - `ComposerBar` 内部维护输入框焦点、发送/停止按钮、`Command+Return` 和辅助语义。
  - `ChatBubbleLayoutPolicy`、`PromptTemplateGridLayoutPolicy` 和 `OptimizationToggleGridLayoutPolicy` 是可测试 UI 宽屏策略参考。
- `LocalGemma/AppState.swift`
  - `InferenceEngine.send`、`stop`、`applyTemplate` 和 `useTemplate` 控制输入和生成状态，本轮不得改动。
- `LocalGemmaTests/LocalGemmaTests.swift`
  - 当前测试基线为 67 个 XCTest 方法。
  - 已有 `testComposerInputMetadataAndFocusPolicyDescribeEntryPoints` 锁住 composer 语义和焦点入口。
  - 已有 `testChatBubbleLayoutPolicyAdaptsToWideChatTranscripts` 可作为宽屏策略测试写法参考。

## 实现步骤

1. 在 `ContentView.swift` 中新增 `ComposerBarLayoutPolicy`。
2. 策略建议：
   - 横向 padding 保持 `18`，底部 padding 保持 `12`。
   - 最小可读宽度约 `320`，避免无效宽度计算跌到不可用值。
   - 最大内容宽度约 `760`，让 Mac/iPad 宽屏输入行长接近聊天消息阅读宽度。
   - 提供 `contentWidth(forContainerWidth:)` 供测试锁住窄屏、宽屏和无效宽度 clamp。
3. 在 `ChatWorkspace.chatSurface` 中将 `ComposerBar` 加上最大宽度并居中，再沿用策略 padding。
4. 不移动 `ComposerBar` 内部 `TextField` 和发送/停止按钮，不改变 view identity 和 focus request 触发方式。
5. 在 `LocalGemmaTests.swift` 新增 `testComposerBarLayoutPolicyConstrainsWideComposerInput`，测试函数数从 67 增加到 68。

## 风险和禁止项

- 风险：如果把 `ComposerBar` 重建为条件视图，可能影响 `@FocusState` 和模板填入后的聚焦；本轮只做外层宽度和居中。
- 风险：最大宽度过小会影响长 prompt 编辑，必须用测试锁住 iPhone、iPad 和 Mac 宽度行为。
- 禁止下载模型权重、云端推理、绕过 verified 门禁。
- 禁止把未跑测试写成已通过。
- 禁止验收旧 run 或旧 artifact。
