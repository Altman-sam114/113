# v2.22 / 聊天气泡宽屏宽度策略

## 版本号和分配依据

- 版本号：`v2.22`
- 分配依据：`update_log.md` 最新正式版本为 `v2.21 / 提示词模板宽屏布局策略`，本轮继续递增到 `v2.22`。
- 所属总目标：Agent X 持续优化 UI、Mac 和 iPad 体验。

## 背景

推理页的 `ChatBubble` 目前把用户消息固定限制为 `310` pt，assistant 和 system 消息则可以铺满可用宽度。这个策略在 iPhone 上可用，但在 iPad 和 Mac Catalyst 宽窗口中，用户长 prompt 会显得过窄，assistant 长回复又缺少最大阅读宽度约束。需要把聊天气泡宽度抽成可测试策略：窄屏保持紧凑，宽屏随容器增长，同时为本地模型和系统消息设置稳定阅读上限。

## 目标

1. 新增 `ChatBubbleLayoutPolicy`，按容器宽度和消息角色计算气泡最大宽度。
2. 让 `ChatTranscript` 获取可用宽度并传给 `ChatBubble`，不修改消息、会话或 runtime 状态流。
3. 用户消息在 iPad/Mac 宽区域从旧 `310` 上限增长到更合适的宽度，但保留最大上限，避免横跨整屏。
4. assistant/system 消息在宽屏下限制阅读宽度，避免超长文本行。
5. 新增 XCTest 锁住 iPhone、iPad 和 Mac 宽窗口下不同角色的宽度策略。
6. 同步 `AGENTS.md`、`README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和 `update_log.md`。

## 非目标

- 不修改 `InferenceEngine`、`ChatMessage` 数据结构、会话创建/删除/导出逻辑。
- 不修改 `ChatMessageAccessibilityMetadata` 或 `ChatTranscriptAccessibilityMetadata` 的文案和 identifier。
- 不修改 composer 输入框、发送/停止按钮、焦点策略或 `Command+Return` 行为。
- 不修改 `SessionSidebarLayoutPolicy`、工作区布局断点或模型页布局。
- 不新增 UI Test target。
- 不下载模型权重，不接入云端推理，不接入真实 runtime，不创建原生 macOS target。

## 当前架构依据

- `LocalGemma/ContentView.swift`
  - `ChatTranscript` 使用 `ScrollViewReader`、`ScrollView` 和 `LazyVStack` 渲染消息列表。
  - `ChatBubble` 当前通过 `.frame(maxWidth: message.role == .user ? 310 : .infinity, ...)` 控制宽度。
  - 单条消息和聊天记录容器已有辅助语义，本轮只改视觉宽度策略，不改辅助语义。
- `LocalGemmaTests/LocalGemmaTests.swift`
  - 当前测试基线为 65 个 XCTest 方法。
  - 已有聊天消息和聊天记录容器辅助语义测试，但没有覆盖聊天气泡宽屏宽度策略。

## 实现步骤

1. 在 `ContentView.swift` 中新增 `ChatBubbleLayoutPolicy`。
2. 策略建议：
   - `minimumReadableWidth` 约 `280`，保证窄屏可用。
   - 用户消息最大宽度约 `520`，按容器宽度比例增长，替代固定 `310`。
   - assistant/system 最大宽度约 `680`，在 Mac/iPad 宽窗口中保留可读行长。
   - `maxWidth(for:availableWidth:)` 对极窄或无效宽度做 clamp，返回不小于最小可读宽度、不大于对应角色最大宽度和容器宽度。
3. 用 `GeometryReader` 或等价方式让 `ChatTranscript` 获取消息列表可用宽度，并将其传给 `ChatBubble`。
4. 修改 `ChatBubble` 接口，新增 `availableWidth` 参数，继续保留现有对齐、背景、辅助语义和 token 显示。
5. 在 `LocalGemmaTests.swift` 新增 `testChatBubbleLayoutPolicyAdaptsToWideChatTranscripts`，测试函数数从 65 增加到 66。

## 风险和禁止项

- 风险：过宽的用户气泡会降低聊天左右对话感，必须保留用户消息最大宽度和右对齐。
- 风险：assistant/system 宽度上限不能让 iPhone 或窄 split view 文本过窄，必须测试窄屏 clamp。
- 禁止下载模型权重、云端推理、绕过 verified 门禁。
- 禁止把未跑测试写成已通过。
- 禁止验收旧 run 或旧 artifact。
