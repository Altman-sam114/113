## 版本分配依据

- 人工明确指定本轮为 Agent X `v0.9`。

- `update_log.md` 当前记录 v0.8 后遗留事项为：Mac 方向仍是后续目标；启用 Mac Catalyst 或独立 macOS target 前，需要单独审计 `UIKit`、`PhotosPicker`、分享、剪贴板和文件导入路径。
- 因此本轮从 v0.8 继续递增为 v0.9，commit 主题必须以 `v0.9:` 开头，CI artifact 版本也应由 commit 主题自动提取为 `v0.9`。

## 背景

当前 App 是 SwiftUI iOS 原型，已经支持 iPhone+iPad 和基于容器尺寸的双栏布局。v0.8 已完成 iPad 大屏准备度，但工程仍没有 Mac 独立 target 或 Mac Catalyst target。

已核对的当前事实：

- `LocalGemma.xcodeproj/project.pbxproj` 中 app/test target 当前 `TARGETED_DEVICE_FAMILY = "1,2"`，`SDKROOT = iphoneos`，没有 `SUPPORTS_MACCATALYST`、`SUPPORTED_PLATFORMS` 或 Mac 专用 target。
- `LocalGemma/ContentView.swift` 目前导入 `UIKit`、`PhotosUI`、`SwiftUI` 和 `UniformTypeIdentifiers`，平台敏感点包括 `UIImage`、`UIGraphicsImageRenderer`、`Image(uiImage:)`、`UIPasteboard`、`ShareLink`、`.fileImporter` 和 `PhotosPicker`。
- `WorkspaceLayoutMode.resolve(for:)` 当前以容器尺寸判断单栏、compact 双栏和 regular 大屏双栏；已有 iPhone 横屏、iPad Pro 竖屏和大屏窗口测试。
- `LocalGemmaTests/LocalGemmaTests.swift` 当前包含 33 个 XCTest 方法。
- `.github/workflows/ci-results.yml` 当前只运行静态检查、LogicSmoke、iOS Simulator `build-for-testing` 和可用 iPhone Simulator `test-without-building`；manifest 只有 iOS build/test outcome，没有 Mac / Catalyst outcome、log 或 result bundle 字段。
- `README.md` 当前明确写着“当前没有 Mac 独立 target 或 Mac Catalyst target”。

本轮目标是为 Mac 方向建立最小可验证基线。首选是启用并验证 Mac Catalyst `build-for-testing`；如果实际工程或 GitHub runner 更适合先验证 Designed for iPad on Mac，则必须把边界写清楚，并禁止把它描述成原生 macOS App 或真实 Mac native target。

## 目标

1. 优先建立 Mac Catalyst 最小可验证基线：
   - 在不创建新 scheme、不创建独立 macOS target 的前提下，启用必要的 Mac Catalyst build setting。
   - 让 `LocalGemma` scheme 可以在 Mac Catalyst destination 下执行 `build-for-testing`。
   - CI 结果包必须记录 Mac Catalyst build outcome、log path 和 result bundle path。
2. 如果经预检确认本轮更适合先验证 Designed for iPad on Mac：
   - 明确选择该路径的具体原因。
   - 不启用或不宣称 Mac Catalyst 已通过。
   - CI manifest 必须把 Catalyst build 标记为 `skipped` 或等价状态，并记录 `macBaselineKind = designed-for-ipad-on-mac`、边界说明和验证日志。
   - 文档必须明确 Designed for iPad on Mac 不是原生 macOS target，也不是 Mac Catalyst 通过。
3. 补充桌面窗口尺寸布局测试，锁住 Mac / 大屏窗口下仍进入 regular 大屏双栏，不破坏 iPhone/iPad 行为。
4. 同步 CI 结果包结构：
   - manifest 增加 Mac / Catalyst baseline 字段。
   - artifact 包增加 Mac / Catalyst build log、baseline notes 和必要 result bundle。
   - failure summary 和 JUnit 覆盖 Mac baseline outcome。
5. 同步文档：
   - `README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、`update_log.md` 必须说明 v0.9 的 Mac 方向边界。
   - 如入口规则当前状态需要更新，同步 `AGENTS.md`。
6. 完成后 commit 到 `main` 并 `git push origin main`，由 GitHub Actions 生成最新 v0.9 未加密结果包，交给 Agent C 下载验收。

- 不把 Designed for iPad on Mac 描述成原生 macOS。
- 不接入真实 Gemma runtime、Core ML / MLX / llama.cpp 推理或外部模型服务。
- 不下载 Gemma 或任何模型权重，不新增大体积测试数据。
- 不改变 artifact `missing` / `staged` / `verified` 门禁。

- 不新增 PR 流程，不使用非 `main` 分支作为云端验证主线。


### 5. 补桌面窗口布局测试

在 `LocalGemmaTests/LocalGemmaTests.swift` 中补充桌面窗口尺寸测试。最低覆盖：

- `CGSize(width: 1280, height: 800)` 或 `CGSize(width: 1440, height: 900)` -> `.landscapeRegular`，`usesSidebar == true`。
- `CGSize(width: 1024, height: 768)` -> `.landscapeRegular`，验证常见 Mac / iPad 窗口尺寸进入大屏双栏。
- 一个窄桌面窗口，例如 `CGSize(width: 760, height: 720)` -> `.landscapeCompact`，验证宽度达到双栏但未达到 regular 时不误判。
- 一个窄窗口，例如 `CGSize(width: 680, height: 900)` -> `.portrait`，验证窄 Mac 窗口或分屏仍可退回单栏。
- `landscapeRegular.sidebarWidth(for:)` 在桌面宽窗口仍保持当前约束，不挤压右侧工作区。

### 6. 更新 CI workflow

修改 `.github/workflows/ci-results.yml`，在现有静态检查、LogicSmoke、iOS build/test 之外，增加 Mac baseline 步骤。

路径 A 时，CI 应运行 Mac Catalyst build-for-testing：

```sh
xcodebuild \
  -project "$XCODE_PROJECT" \
  -scheme "$SCHEME" \
  -configuration Debug \
  -sdk macosx \
  -destination 'platform=macOS,variant=Mac Catalyst' \
  -derivedDataPath "$DERIVED_DATA_PATH-catalyst" \
  -resultBundlePath "$CI_RESULTS_DIR/LocalGemma-maccatalyst-build.xcresult" \
  CODE_SIGNING_ALLOWED=NO \
  build-for-testing >"$CI_RESULTS_DIR/mac-catalyst-build.log" 2>&1
```

路径 B 时，CI 应生成 `mac-baseline-notes.md` 和对应 log，记录 Designed for iPad on Mac 的 build setting / showBuildSettings 结果，并把 Catalyst build 标记为 skipped。

manifest 至少新增这些字段或等价字段：

```json
{
  "macBaselineKind": "mac-catalyst | designed-for-ipad-on-mac",
  "macCatalystBuildOutcome": "success | failure | skipped",
  "macCatalystDestination": "platform=macOS,variant=Mac Catalyst | not-run",
  "macCatalystBuildLogPath": "ci-results/mac-catalyst-build.log",
  "macCatalystResultBundlePath": "ci-results/LocalGemma-maccatalyst-build.xcresult",
  "macCatalystSkippedReason": "",
  "macDesignedForIPadOutcome": "success | failure | skipped",
  "macBaselineNotesPath": "ci-results/mac-baseline-notes.md"
}
```

JUnit 和 `ci-failure-summary.md` 必须同步体现 Mac baseline：

- 路径 A：Mac Catalyst build 是必过项；`macCatalystBuildOutcome != success` 时 CI 应失败。
- 路径 B：Designed for iPad on Mac baseline 是必过项；Catalyst skipped 只有在 notes 写明原因时才可接受。
- 无论哪条路径，都不能让 manifest 缺字段或把未运行的 Catalyst 写成 success。

结果包仍必须保持小体积，只上传 manifest、artifact-name、JUnit、关键日志、失败摘要和必要 `.xcresult`，禁止上传 DerivedData、模型、缓存或历史 artifact。

## Agent C 验收标准

Agent C 必须只验收最新 `origin/main` 对应的 v0.9 结果包：

1. 执行 `git fetch origin`，核对 `main`、`origin/main` 和 Agent B 输出的 commit SHA。
2. 执行 `gh run list --workflow ci-results.yml --branch main --limit 5`，确认最新 run 对应 v0.9 commit。
3. 下载最新 run artifact 到 `/private/tmp/localgemma-c-review-<run_id>/`。
4. 核对 `ci-artifact-manifest.json`：
   - `branch == main`
   - `version == v0.9`
   - `commitSha == origin/main 最新 commit`
   - `commitSubject` 以 `v0.9:` 开头
   - `artifactName` 等于 `artifact-name.txt`
   - `runId`、`runAttempt` 与下载 run 一致
   - `staticChecksOutcome == success`
   - `logicSmokeOutcome == success`
   - `buildOutcome == success`
   - `testOutcome == success`
   - `macBaselineKind` 与 Agent B 选择路径一致
   - 路径 A：`macCatalystBuildOutcome == success`，`macCatalystDestination` 指向 Mac Catalyst，log 和 result bundle 存在
   - 路径 B：`macCatalystBuildOutcome == skipped`，`macCatalystSkippedReason` 非空，`macDesignedForIPadOutcome == success`，`mac-baseline-notes.md` 明确不是原生 macOS / 不是 Catalyst 通过
5. 打开 `ci-failure-summary.md`、`junit.xml`、`static-checks.log`、`logic-smoke.log`、`xcodebuild.log`、`test.log`、Mac baseline log / notes 和 `.xcresult` 或等价结果，确认没有隐藏失败。
6. 复核 diff 和文档：
   - Mac baseline 选择路径与工程设置、CI manifest 和 README 一致。
   - 如果宣称 Mac Catalyst，则必须有实际 Catalyst build-for-testing 证据。
   - 如果只是 Designed for iPad on Mac，则文档没有把它写成原生 macOS 或 Catalyst passed。
   - 桌面窗口布局测试已补充，且 iPhone/iPad 断点未回退。
   - 没有真实 runtime、模型下载、云端推理、PR 流程或非 main 分支改动。
7. 如果任一 required outcome 不是 success，或结果包不是最新 `origin/main`，或文档/源码/manifest 互相矛盾，则验收不通过，退回 Agent B 在 `main` 上追加修复 commit。

Agent C 通过时应输出版本号、commit hash、run id、run attempt、artifact 名称、已核对文件、Mac baseline 类型和建议下一步。

## 风险

- Mac Catalyst build setting 可能需要 `SUPPORTED_PLATFORMS`、`TARGETED_DEVICE_FAMILY` 或 test host 配置同步；写错会导致 iOS build 或 test target 失败。
- `UIKit`、`PhotosPicker`、`ShareLink`、`UIPasteboard`、`UIImage` 和 `UIGraphicsImageRenderer` 在 Catalyst / Designed for iPad on Mac 下的可用性要以实际 build 为准，不能凭文档措辞认定通过。
- CI runner 的 Xcode 版本和 Catalyst destination 支持可能变化，应以 `environment.log`、Mac build log 和 manifest 为准。
- 如果选择路径 B，最大风险是后续误读为 Catalyst 或原生 macOS 已完成；必须用字段和文档把边界锁死。
- 增加 Mac result bundle 会让 artifact 变大；只能上传必要 `.xcresult` 和日志，不上传 DerivedData 或缓存。
- 修改 CI manifest 后，Agent C 的验收字段也必须同步更新，否则容易错验旧字段。

## 禁止项

- 禁止下载模型权重。
- 禁止接入云端推理、外部模型 API、MLX、llama.cpp 或真实 Core ML 推理。
- 禁止绕过 artifact `verified` 门禁。
- 禁止把 `manual-import-required` artifact 当成真实可运行权重。
- 禁止把 `RealGemmaRuntimePlaceholder` 描述成真实推理已完成。
- 禁止创建原生 macOS target 或宣称已有原生 macOS App。
- 禁止把 Designed for iPad on Mac 写成 Mac Catalyst 通过。
- 禁止在未实际运行 Catalyst build-for-testing 时写 `macCatalystBuildOutcome == success`。
- 禁止使用非 `main` 分支作为提交和云端验证主线。
- 禁止创建 PR 或等待 PR merge。
- 禁止伪造测试结果、commit SHA、push 结果、run id、artifact 名称或 Agent C 验收结论。
- 禁止回滚、删除或覆盖用户/其他 Agent 的未提交改动。
- 禁止无关重构、无关 UI 改版或大文件拆分。
- 禁止提交模型、大数据、DerivedData、证书、密码、token 或 secret。
