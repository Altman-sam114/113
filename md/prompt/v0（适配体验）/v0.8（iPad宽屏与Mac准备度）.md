## 背景

当前 App 是 SwiftUI iOS 原型，产品叙述仍偏向 iPhone，但下一轮目标是把运行体验扩展到 iPhone+iPad，并为后续 Apple Silicon / Mac 方向留下清晰边界。

已核对的当前事实：

- `LocalGemma.xcodeproj/project.pbxproj` 中 app target 和 test target 的 Debug / Release 配置均为 `TARGETED_DEVICE_FAMILY = 1;`，目前只声明 iPhone。
- `LocalGemma/ContentView.swift` 的 `WorkspaceLayoutMode.resolve(for:)` 当前逻辑是 `size.width > size.height && size.width >= 700` 才进入侧栏布局；因此 iPad Pro 竖屏这类大画布会被判成 `.portrait` 单栏。
- `WorkspaceLayoutMode` 同时被 `ContentView` 主布局和 `ChatWorkspace` 内部会话栏布局复用；断点调整必须避免在窄右侧内容区误开嵌套侧栏。
- `LocalGemmaTests/LocalGemmaTests.swift` 当前有 32 个 `test...` 方法，已有 `testWorkspaceLayoutModeResolvesLandscapeVariants` 和 `testWorkspaceLayoutModeConstrainsSidebarWidth`，但缺少 iPad Pro 竖屏、大屏窗口和 target family 变更后的文档基线。
- `README.md`、`md/flow/flow.md` 和 `md/flow/flowchart.md` 当前仍以“iPhone 端侧部署”和“横屏布局”为主描述；v0.8 后应改成 iPhone+iPad，并说明双栏断点不再只等同物理横屏。

## 目标

1. 将 app target 和 test target 扩展为 iPhone+iPad：
   - `LocalGemma` Debug / Release：`TARGETED_DEVICE_FAMILY = "1,2";`
   - `LocalGemmaTests` Debug / Release：`TARGETED_DEVICE_FAMILY = "1,2";`
2. 改进 `WorkspaceLayoutMode.resolve(for:)`，让 iPad Pro 竖屏和足够大的窗口进入双栏大屏布局。
3. 保留 iPhone 行为：
   - iPhone 竖屏仍为单栏 `.portrait`。
   - iPhone 横屏仍可进入 `.landscapeCompact`。
   - 现有宽横屏仍可进入 `.landscapeRegular`。
4. 补充布局单元测试，锁住 iPad Pro 竖屏、大屏窗口、iPhone 竖屏和 iPhone 横屏断点。
5. 同步文档：`AGENTS.md`、`README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、`update_log.md`，必要时同步 `md/prompt/README.md` 中与本轮流程相关的事实。
6. 通过 main 直推触发 GitHub Actions，由 Agent C 下载最新未加密结果包验收。

## 非目标

- 不创建 macOS 独立 target、Mac Catalyst target、visionOS target 或新的 Xcode scheme。
- 不接入真实 Gemma runtime、Core ML / MLX / llama.cpp 推理或外部模型服务。
- 不下载 Gemma 或任何模型权重，不新增大体积测试数据。
- 不改变 artifact `missing` / `staged` / `verified` 门禁。
- 不把 `manual-import-required` 视为真实可运行权重。
- 不重做 UI 视觉风格，不拆分 `ContentView.swift` / `AppState.swift` 大文件，除非为编译或测试最小修复所必需。
- 不新增 UI Test target；本轮用现有 XCTest + 云端 build/test 结果包验证。

### 3. 改进 WorkspaceLayoutMode 断点

在 `LocalGemma/ContentView.swift` 中调整 `WorkspaceLayoutMode.resolve(for:)`，目标行为：

- `CGSize(width: 390, height: 844)` -> `.portrait`
- `CGSize(width: 430, height: 932)` -> `.portrait`
- `CGSize(width: 844, height: 390)` -> `.landscapeCompact`
- `CGSize(width: 1024, height: 1366)` -> `.landscapeRegular`
- `CGSize(width: 1000, height: 760)` -> `.landscapeRegular`
- `CGSize(width: 1180, height: 820)` -> `.landscapeRegular`

建议实现方向：

- 用宽度判断是否能容纳主侧栏，例如 `width >= 700` 才允许双栏。
- 用宽度和高度共同判断 regular 大屏，例如 `width >= 980 && height >= 700`。
- 对 iPhone 横屏这类高度较低但宽度足够的场景继续返回 `.landscapeCompact`。
- 对 iPad 窄分屏或中等竖屏窗口，可以返回 `.landscapeCompact`，但不要让 iPhone 竖屏误进双栏。
- 避免引入 `UIDevice.current.userInterfaceIdiom`、`UIScreen.main.bounds` 或设备型号判断；Stage Manager、分屏和未来 Mac 窗口准备度都应由实际容器尺寸驱动。

Agent B 可以提炼少量私有静态阈值常量，便于测试和文档说明；不要做无关的大文件拆分。

- `WorkspaceLayoutMode` 的 case 名称带 `landscape`，但 v0.8 后会覆盖 iPad 竖屏大屏场景；如果不更新文档，后续 Agent 容易误解断点语义。
- 主 `ContentView` 和 `ChatWorkspace` 复用同一个 resolver；阈值过低会在右侧窄内容区打开内部会话栏，阈值过高会让 iPad 竖屏继续单栏。
- CI 默认只选 iPhone simulator；iPad 支持要靠 target family、尺寸驱动单元测试和 Xcode build/test 结果包共同证明。
- Xcode project 的 `TARGETED_DEVICE_FAMILY` 格式如果写错，会导致 `plutil` 或 Xcode build 失败。
