## 背景

当前项目是 SwiftUI iOS 原型 App，已支持 iPhone、iPad 与 Mac Catalyst `build-for-testing` 基线。v0.9 的 CI 已覆盖 iOS Simulator build/test、Mac Catalyst `build-for-testing`、manifest Mac baseline 字段和小体积未加密结果包。

已核对的当前事实：

- 工程形态是 Xcode iOS 工程：存在 `LocalGemma.xcodeproj`，没有 `Package.swift`，没有原生 macOS target。
- `LocalGemma.xcodeproj/project.pbxproj` 中 app/test target 已启用 `SUPPORTS_MACCATALYST = YES`，`TARGETED_DEVICE_FAMILY = "1,2"`，iOS deployment target 为 17.0。
- `LocalGemma/LocalGemmaApp.swift` 只有一个 `WindowGroup`，创建 `ModelCatalog(autoScanLocalArtifacts: true)`、`InferenceEngine()` 和 `DeviceOptimizer()` 并注入 `ContentView`。
- `LocalGemma/ContentView.swift` 通过 `WorkspaceLayoutMode.resolve(for:)` 按容器尺寸切换单栏、compact 双栏和 regular 大屏双栏；`ChatWorkspace`、`ModelLibraryView`、`ExportSessionView`、`ComposerBar` 分别覆盖会话、文件导入、分享/复制和输入发送。
- `LocalGemmaTests/LocalGemmaTests.swift` 当前包含 34 个 XCTest 方法，已覆盖 Mac Catalyst/桌面窗口尺寸布局断点，但尚未覆盖 v1.0 的“本地 Mac Catalyst run 入口”契约。
- `.github/workflows/ci-results.yml` 当前已有 `macCatalystBuildOutcome`、`macCatalystDestination`、`macCatalystBuildLogPath`、`macCatalystResultBundlePath` 和 `macBaselineNotesPath` 等字段，但没有检查 `script/build_and_run.sh` 的存在性、可执行权限和语法，也没有 manifest 字段记录本地 run 入口。
- 当前仓库没有 `script/` 目录，也没有 `.codex/environments/environment.toml`。

本轮目标不是再证明 Catalyst 能编译，而是在 v0.9 基线上建立项目内可重复的 Mac Catalyst build/run 入口，并把桌面窗口与交互边界写进测试、CI 契约和文档，方便后续 Agent 和人工在同一入口验证。

## 目标

1. 建立项目内可重复的 Mac Catalyst 本地 build/run 入口：
   - 新增 `script/build_and_run.sh`，放在项目根的 `script/` 目录，不放入 Swift app 源码目录。
   - 脚本面向当前 Xcode project，使用 `xcodebuild` 构建 Mac Catalyst Debug app，并通过 `/usr/bin/open -n` 启动生成的 `.app`。
   - 脚本默认路径应是 kill 旧进程、build、launch；同时支持 `--verify`、`--logs`、`--telemetry`、`--debug`、`--build-only` 或等价窄模式。
   - 脚本必须使用项目内 DerivedData，例如 `.build/DerivedDataCodex-MacCatalystRun`，不要写入无关目录。
2. 可选建立 Codex Run action：
   - 如 Agent B 选择添加，写入 `.codex/environments/environment.toml`，Run action 必须指向 `./script/build_and_run.sh`。
   - 如 Agent B 选择暂不添加，必须在 README 或测试规范说明本轮只提供 CLI run 入口，且 CI manifest 对 environment check 标记为 `skipped` 并写清原因。
3. 做 Mac Catalyst 桌面窗口与交互窄审计：
   - 复核启动窗口、大屏/窄窗口布局、滚动容器、输入框发送、导出分享/复制、文件导入、壁纸选择等 Catalyst 敏感边界。
   - 只有发现明确问题时才做极小 UI 修正；允许范围包括 Catalyst-only 最小窗口尺寸、键盘快捷键、滚动/分享/菜单边界、或不改变视觉风格的可用性修复。
   - 不重做视觉风格，不大拆 `ContentView.swift` 或 `AppState.swift`。
4. 增加能够锁住 v1.0 基线的测试或可验证契约：
   - 最低必须让 CI 静态检查 `script/build_and_run.sh` 存在、可执行、`bash -n` 通过。
   - 如做 Swift UI/状态窄改，必须新增或修改对应 XCTest，测试数量和文档同步更新。
   - 如果现有 `WorkspaceLayoutMode` 测试已覆盖需要的窗口尺寸，不要为了增加数量而重复无意义测试；可以用 CI 脚本契约和文档作为 v1.0 的主要可验证产物。
5. 同步 CI 结果包结构：
   - manifest 增加本地 Mac Catalyst run 入口相关字段。
   - `junit.xml`、`ci-failure-summary.md` 和结果包日志要覆盖 run script 契约检查。
6. 同步核心文档：
   - `README.md`、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`update_log.md` 必须说明 v1.0 的本地 run 入口、Mac Catalyst 窗口/交互基线和非目标边界。
   - 如入口规则当前状态需要更新，同步 `AGENTS.md`。
7. 完成后 commit 到 `main` 并 `git push origin main`，由 GitHub Actions 生成最新 v1.0 未加密结果包，交给 Agent C 下载验收。

### 3. 新增 `script/build_and_run.sh`

新增文件：

```text
script/build_and_run.sh
```

脚本要求：

- `#!/usr/bin/env bash`
- `set -euo pipefail`
- `ROOT_DIR` 必须从脚本路径解析到项目根。
- `DEVELOPER_DIR` 如未设置，默认使用 `/Applications/Xcode.app/Contents/Developer`。
- `APP_NAME="LocalGemma"`，`SCHEME="LocalGemma"`，`PROJECT="$ROOT_DIR/LocalGemma.xcodeproj"`。
- `DERIVED_DATA_PATH="$ROOT_DIR/.build/DerivedDataCodex-MacCatalystRun"` 或等价项目内路径。
- 默认 destination 优先使用 `platform=macOS,variant=Mac Catalyst`；如本机只接受 `generic/platform=macOS,variant=Mac Catalyst` 用于 build-only，脚本必须区分可运行 build 和 generic build，不能把 generic build-for-testing 写成已 launch。
- build 后应在 DerivedData products 下定位 `LocalGemma.app`，例如搜索 `Build/Products/*maccatalyst*/LocalGemma.app`，避免硬编码一个未来 Xcode 可能变化的完整路径。
- 默认 `run` 模式：
  1. `pkill -x "$APP_NAME"`，不存在则忽略。
  2. 使用 `xcodebuild ... build` 构建 Mac Catalyst Debug app。
  3. 用 `/usr/bin/open -n "$APP_BUNDLE"` 启动。
- `--verify` 模式：启动后短暂等待，用 `pgrep -x "$APP_NAME"` 或等价方式确认进程存在；如果窗口服务器、签名或沙箱阻止 launch，必须让命令失败并输出明确原因。
- `--build-only` 模式：只构建并打印 app bundle 路径，供无 GUI 环境快速验证。
- `--logs` 模式：启动后运行 `/usr/bin/log stream --info --style compact --predicate "process == \"LocalGemma\""` 或等价进程过滤。
- `--telemetry` 模式：如当前 app 没有专用 Logger subsystem，可先使用 process predicate；不要为了 telemetry 大改业务代码。
- `--debug` 模式：构建后用 `lldb -- "$APP_BUNDLE/Contents/MacOS/LocalGemma"` 或等价方式启动/调试。
- `--help` 或未知参数应输出用法并退出非 0。
- 文件必须设置可执行权限。

注意：

- 如果本机 Mac Catalyst launch 需要签名设置，优先用最小脚本参数或 Xcode build setting 解决；不得提交个人证书、profile、team secret 或账号信息。
- `CODE_SIGNING_ALLOWED=NO` 适合 CI build-for-testing，但可能不适合本地 launch。Agent B 必须以实际可运行 bundle 为准，不能只 build-for-testing 后宣称已 run。
- 脚本不得下载模型、访问外部推理服务、写入模型目录或修改用户 artifact。

### 6. 测试与 CI 契约

最低必须修改 `.github/workflows/ci-results.yml`，让 CI 静态检查 run script 契约：

```sh
test -f script/build_and_run.sh
test -x script/build_and_run.sh
bash -n script/build_and_run.sh
```

建议新增一个独立日志：

```text
ci-results/mac-catalyst-run-script.log
```

manifest 建议新增字段：

```json
{
  "macCatalystRunEntrypoint": "script/build_and_run.sh",
  "macCatalystRunScriptCheckOutcome": "success/failure",
  "macCatalystRunScriptLogPath": "ci-results/mac-catalyst-run-script.log",
  "codexRunEnvironmentPath": ".codex/environments/environment.toml",
  "codexRunEnvironmentCheckOutcome": "success/skipped/failure",
  "codexRunEnvironmentSkippedReason": ""
}
```

规则：

- `macCatalystRunScriptCheckOutcome` 是必过项；如果脚本不存在、不可执行或 `bash -n` 失败，CI 必须失败。
- 如果 `.codex/environments/environment.toml` 存在，CI 应检查包含 `command = "./script/build_and_run.sh"`；失败则 `codexRunEnvironmentCheckOutcome == failure`。
- 如果 `.codex/environments/environment.toml` 不存在，CI 可写 `codexRunEnvironmentCheckOutcome == skipped`，但 `codexRunEnvironmentSkippedReason` 必须非空，例如 `not-added-in-v1.0-cli-entrypoint-only`。
- `junit.xml` 的 tests/failures/skipped 计数必须包含 run script 契约；`ci-failure-summary.md` 必须列出脚本检查 outcome。
- 继续保留 v0.9 已有 Mac Catalyst build-for-testing step，不要用脚本检查替代 Catalyst build。

Swift 测试要求：

- 如果没有 Swift UI/状态代码改动，可以不新增 XCTest，但必须保留 34 个测试函数，且文档说明 v1.0 的新增验证点主要是 shell run entrypoint 和 CI manifest contract。

- 如新增 Swift 文件，必须加入 Xcode target；本轮应尽量避免新增 Swift 文件。

## Agent C 验收标准

4. 核对 `ci-artifact-manifest.json`：
   - `branch == main`
   - `version == v1.0`
   - `commitSha == origin/main 最新 commit`
   - `commitSubject` 以 `v1.0:` 开头
   - `artifactName` 等于 `artifact-name.txt`
   - `runId`、`runAttempt` 与下载 run 一致
   - `staticChecksOutcome == success`
   - `logicSmokeOutcome == success`
   - `buildOutcome == success`
   - `testOutcome == success`
   - `macBaselineKind == mac-catalyst`
   - `macCatalystBuildOutcome == success`
   - `macCatalystDestination` 指向 Mac Catalyst
   - `macCatalystRunEntrypoint == script/build_and_run.sh`
   - `macCatalystRunScriptCheckOutcome == success`

   - 如果 `codexRunEnvironmentCheckOutcome == skipped`，`codexRunEnvironmentSkippedReason` 必须非空且文档说明未添加 Run action 的原因
5. 打开 `ci-failure-summary.md`、`junit.xml`、`static-checks.log`、`logic-smoke.log`、`xcodebuild.log`、`test.log`、`mac-catalyst-build.log`、`mac-catalyst-run-script.log`、`mac-baseline-notes.md` 和 `.xcresult` 或等价结果，确认没有隐藏失败。
6. 复核 diff 和文档：
   - `script/build_and_run.sh` 位于项目根 `script/`，可执行，内容使用 Xcode project + Mac Catalyst，不是 SwiftPM GUI 模板。
   - 脚本不下载模型、不接外部推理、不写入模型 artifact 目录。

   - 没有创建原生 macOS target，没有真实 runtime、模型下载、云端推理、PR 流程或非 main 分支改动。

## 风险

- Mac Catalyst `build` 与 `build-for-testing` 的签名和产物路径可能不同；脚本必须以实际可运行 `.app` 为准，不能把 build-for-testing 结果冒充 launch 成功。
- Codex 沙箱或 macOS 窗口服务器可能阻止 `/usr/bin/open`、`pgrep` 或 log stream；失败时要记录真实错误，并把 GUI run 标为未完成或失败。
- Xcode 版本可能改变 DerivedData products 子目录名；脚本应搜索 app bundle，而不是硬编码唯一产物路径。
- `.codex/environments/environment.toml` 是本地 Codex app 配置入口，不能写入个人绝对路径、secret 或机器专属配置。
- 对 `ContentView` 做 Catalyst-only 修正时，容易误伤 iPhone/iPad 布局；必须保持 `WorkspaceLayoutMode` 测试和现有行为。
- CI 不能真实启动 GUI app；Agent C 只能验收云端 build/test/script contract，本地窗口实际表现仍需要 Agent B 如实报告可选 run 验证。
- 增加 manifest 字段后，`junit.xml`、failure summary、README 和 `md/test/test.md` 必须同步，否则 Agent C 容易错验旧结构。

- 禁止在未实际运行 `./script/build_and_run.sh --verify` 时写成“已启动并验证窗口”。
- 禁止在 CI 中伪造 GUI launch 成功。
- 禁止使用非 `main` 分支作为提交和云端验证主线。
- 禁止创建 PR 或等待 PR merge。
