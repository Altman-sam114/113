## 背景

当前项目是 SwiftUI iOS 工程，已支持 iPhone、iPad 与 Mac Catalyst build/run 基线。`ContentView.swift` 中的四个 workspace 已通过 `WorkspaceTab.shortcutKey` 暴露 `Command+1...4`，现有按钮也保留了可见导航路径。

Mac Catalyst 和 iPad 外接键盘用户还会从系统菜单发现 app 操作。SwiftUI 的 scene-level `.commands` 和 `CommandMenu` 可以提供原生命令入口，但不应引入 AppKit responder chain 或扩大状态层。当前 `selectedTab` 是 `ContentView` 的局部 UI 状态，最小方案是用 `FocusedValues` 把 `Binding<WorkspaceTab>` 暴露给 app scene 命令层，让菜单项只切换 workspace。

## 目标

1. 增加一个 app-specific command menu，例如 `工作区`。
2. 菜单包含四个工作区切换命令，顺序固定：
   - 推理：`Command+1`
   - 模型：`Command+2`
   - 提示词：`Command+3`
   - 设置：`Command+4`
3. 保持 v1.1 已有快捷键不变：
   - `Command+1...4` 继续切换工作区。
   - `Command+N` 继续新建会话。
   - `Command+Shift+E` 继续导出会话。
   - `Command+Return` 继续发送或停止。
4. 用 XCTest 锁住 command menu 的映射元数据：
   - `WorkspaceTab` 提供可测试的菜单标题或 command title 属性。
   - 新增测试确认 command 顺序覆盖四个 workspace、目标 tab 正确、shortcutKey 未变化。
5. 同步文档和 CI 说明：
   - README、测试规范、flow、flowchart、update_log 必须说明 v1.2 的 Mac/iPad 命令菜单发现层。
   - 测试数量从 35 增加到 36；如果最终数量不同，以实际 `grep -n "func test"` 为准并同步。
6. 完成后 commit 到 `main` 并 `git push origin main`，由 GitHub Actions 生成最新 v1.2 未加密结果包，交给 Agent C 下载验收。

## 非目标

- 不引入 AppKit responder chain、menu item validation 或复杂命令状态。
- 不把导出、新建会话、发送/停止等会话操作搬进 scene command；本轮只做 workspace command menu。
- 不重做视觉风格，不新增可见快捷键说明文本。
- 不改变 `InferenceEngine`、`ModelCatalog`、artifact validation、runtime plan 或真实模型门禁。
- 不下载模型权重，不接云端推理，不新增大体积测试数据。
- 不删除 v1.1 按钮上的 `keyboardShortcut`；命令菜单只是发现层和备用入口。

## 风险

- `CommandMenu` 与按钮同时注册同一个快捷键时，平台可能出现重复命令展示；本轮接受这是菜单发现层目标，但必须保持同一 `WorkspaceTab.shortcutKey` source of truth，避免行为分叉。
- `FocusedValues` 只在当前窗口有焦点上下文时可用；binding 不存在时命令必须禁用或安全无操作。
- SwiftUI `.commands` 在 iOS/iPadOS/Mac Catalyst 编译行为需要 typecheck 和云端 CI 验证；如平台限制出现，做最小条件编译并记录边界。
- 菜单标题不要过长；`工作区` 和四个 workspace 标题应保持短、可扫描。
